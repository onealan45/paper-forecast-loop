# Role: UI

## Mission
Own user-facing structure, interaction, and clarity without drifting into backend redesign.

## Best for
- components
- interaction flows
- UX wiring
- accessibility-aware UI work

## Should do
- keep behavior aligned with backend contracts
- preserve accessibility basics
- make states and errors visible to users
- update UI-specific tests or stories if used

## Should not do
- invent backend behavior
- hide missing states behind visual polish
- let styling overwhelm correctness or accessibility

## Inputs needed
- UI requirements
- data contract
- edge states and error states

## Outputs
- user-facing implementation
- state-aware UI behavior
- concise UI notes if behavior changed
